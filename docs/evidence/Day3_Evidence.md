# Day3 Evidence - Mock 数据建模与列表渲染

## 1. 今日完成内容

今天主要完成了 Mock 数据层搭建和前端列表渲染功能，具体包括：

1. **依赖安装**：安装了 axios（HTTP 请求库）和 json-server（Mock API 服务）
2. **Mock 数据创建**：在项目根目录创建 db.json，包含 4 类业务数据，每类 6 条记录
3. **JSON Server 配置**：在 package.json 中添加 mock 启动命令，Mock 服务运行在 3001 端口
4. **Axios 封装**：创建 src/api/http.ts 基础请求实例，配置 baseURL 和超时时间
5. **业务 API 模块**：按业务模块拆分创建 trade.ts、lostFound.ts、groupBuy.ts、errand.ts 四个 API 文件
6. **通用组件开发**：创建 ItemCard.vue（通用列表卡片）和 EmptyState.vue（空状态提示）组件
7. **页面列表渲染**：在四个核心业务页面（二手交易、失物招领、拼单搭子、跑腿委托）中调用 API 获取数据并使用组件渲染列表

## 2. Mock 数据结构说明

|---|---|---|---|

## 3. 我的设计

在数据字段设计上，我结合校园场景的实际需求进行了思考：

- **二手交易的 price 和 condition**：二手商品最核心的信息就是价格和新旧程度，这直接影响买家的购买决策。price 字段用于排序和筛选，condition（如九成新、八成新）帮助买家快速判断商品状态，避免过多沟通成本。

- **失物招领的 type 字段**：失物招领分为"寻物"（lost）和"招领"（found）两种完全相反的场景，type 字段用于在列表中用不同标签颜色区分（蓝色"寻物启事"vs绿色"失物招领"），让用户一眼识别信息类型。

- **拼单搭子的 targetCount 和 currentCount**：拼单的核心是"人数是否凑齐"，两个数字对比可以直观显示进度（如 3/6 人），还可以计算出还差几人拼成，这是拼单类应用区别于普通信息发布的关键字段。

- **跑腿委托的 from、to 和 reward**：跑腿任务的本质是"帮我把东西从A点送到B点"，所以必须有起点和终点；reward（报酬）是驱动接单的核心因素，用醒目的红色显示在卡片底部，吸引同学接单。

此外，我还为所有数据类型统一添加了 status（open/closed）字段，为后续实现"标记已完成""下架"等功能预留空间；添加了 publisher 字段便于后续扩展用户系统；添加了 publishTime/eventTime/deadline 等时间字段，让信息更有时效性。

## 4. AI 设计

AI 工具在今天的开发中提供了很大帮助，生成的内容包括：

- **db.json 初始结构**：AI 根据任务描述生成了四类数据的基础结构和示例数据，给出了推荐的字段名和数据类型，为后续扩展提供了很好的起点。
- **API 模块代码**：AI 生成了 http.ts 的基础封装和四个业务 API 文件的 TypeScript 接口定义与请求函数，代码结构规范，类型定义完整。
- **页面组件模板**：AI 提供了 TradeView.vue 的完整实现示例，包括模板、脚本和样式，作为其他三个页面的参考模板，大大提高了开发效率。
- **通用组件样式**：AI 生成的 ItemCard 和 EmptyState 组件样式简洁美观，采用了现代 CSS 设计（圆角、阴影、flex 布局）。

AI 生成内容中也有一些需要注意的地方：
- 初始示例数据只有 2 条，不满足"每类不少于5条"的要求，需要人工补充
- 部分字段命名可以更贴合校园场景，需要根据实际业务调整
- 页面样式比较基础，需要人工调整各页面 footer 插槽的差异化展示

## 5. 最终调整

在 AI 生成的基础上，我进行了以下人工修改和优化：

1. **补充数据量**：将每类数据从 2 条补充到 6 条，覆盖更多校园真实场景（如考研资料、毕业转让、快递代取、奶茶拼单等）
2. **调整字段映射**：
   - 失物招领页面将 type 字段映射为中文标签（"寻物启事"/"失物招领"）而非直接显示英文
   - 跑腿委托页面将 from 和 to 组合为"菜鸟驿站 → 西区5栋"的路线格式显示在 location 位置
   - 拼单搭子页面将 deadline 作为时间显示，而非 publishTime
3. **差异化 footer 设计**：
   - 二手交易：显示价格（￥XX）和成色标签
   - 失物招领：显示联系方式（蓝色文字）
   - 拼单搭子：显示人数进度（X/Y 人，绿色文字）
   - 跑腿委托：显示报酬（￥XX，红色醒目文字）
4. **添加空状态判断**：在四个页面都添加了 EmptyState 组件的 v-if/v-else 判断，数据为空时显示友好提示
5. **修正页面标题**：确保各页面 h1 标题与导航栏文案一致（如"拼单搭子"而非"团购拼单"）

## 6. 遇到的问题与解决方法

**问题**：在添加 EmptyState 组件后，浏览器控制台出现 Vue 警告 "Failed to resolve component: EmptyState"，但页面数据正常显示。

**原因分析**：这是 Vite HMR（热模块替换）的缓存问题。虽然我已经在四个页面中正确 import 了 EmptyState 组件，但 HMR 在检测到新组件文件时没有完全重新加载模块，导致旧的模块缓存中没有注册新组件。

**解决方法**：
1. 首先确认代码层面没有问题——import 语句正确、组件文件存在、路径无误
2. 尝试 Ctrl+F5 强制刷新浏览器，清除前端缓存
3. 最终停止开发服务器（Ctrl+C），重新运行 `npm run dev` 完全重启 Vite
4. 重启后在新标签页打开页面，警告消失，组件正常解析

**经验总结**：当遇到"组件无法解析"但代码看起来正确时，大概率是构建工具的缓存问题，重启开发服务器通常能解决。后续开发中如果新增文件后出现类似问题，可以优先尝试重启服务。

## 7. 今日反思

今天的工作让我深刻体会到 Mock 数据建模、接口请求和列表渲染在前端开发中的基础性作用。Mock 数据不仅仅是"随便造点假数据"，好的数据结构设计能够让后续的页面开发事半功倍——字段设计合理，页面渲染就顺畅；字段考虑不周，后面就要反复修改。JSON Server 提供了零配置的 RESTful API 服务，让前端在后端接口就绪之前就能独立开发和调试，前后端可以并行工作，大大提升开发效率。列表渲染则是前端最常见的模式之一，通过封装通用的 ItemCard 组件，我学会了如何通过 props 和插槽（slot）实现组件复用，避免在每个页面重复写相似的卡片结构。

今天的实践也让我理解了"数据驱动视图"的理念——页面长什么样，本质上是由数据结构决定的。先设计好数据模型，再写页面组件，思路会清晰很多。Axios 的模块化封装也让我认识到工程化的重要性，把 API 请求按业务拆分到不同文件，而不是散落在各个组件中，后续维护和修改都会轻松很多。
