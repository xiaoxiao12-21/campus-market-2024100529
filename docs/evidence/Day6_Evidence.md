## 2. 注册登录设计说明

本项目注册/登录的基本流程如下：

- **注册流程**：用户在注册页面填写用户名、密码、姓名、学院、年级后，前端先调用 `getUsers()` 查询所有用户，检查用户名是否已存在；若不存在，则调用 `createUser()` 将新用户数据写入 JSON Server 的 `users` 资源；注册成功后跳转到登录页。
- **登录流程**：用户输入用户名和密码后，`userStore.login()` 调用 `getUsers()` 从 JSON Server 查询所有用户，找到匹配的用户名和密码后，将用户信息保存到 Pinia 的 `currentUser` 中，同时设置 `isLoggedIn = true`，并将用户数据序列化后写入 `localStorage`；登录成功后跳转到个人中心。
- **状态恢复**：应用启动时（`App.vue` 的 `onMounted` 钩子）调用 `userStore.restoreLogin()`，从 `localStorage` 读取用户数据，若存在有效数据则恢复 Pinia 中的登录状态。
- **退出登录**：调用 `userStore.logout()`，清空 Pinia 中的 `currentUser` 和 `isLoggedIn`，同时移除 `localStorage` 中保存的用户数据，然后跳转到登录页。

## 4. 状态持久化说明

为什么需要 localStorage：

- Pinia 作为前端状态管理库，其存储在内存中的状态在页面刷新后会全部丢失，用户每次刷新都需要重新登录，体验很差；
- localStorage 是浏览器提供的持久化存储方案，可以在页面刷新、关闭浏览器重新打开后仍然保留数据；
- 本项目在登录成功后将当前用户信息保存到 localStorage，App 启动时通过 `restoreLogin()` 方法从 localStorage 读取并恢复 Pinia 状态，实现了刷新页面后保持登录的效果；
- 退出登录时同步清空 Pinia 和 localStorage，确保状态一致；
- 需要明确说明：本项目只是 Mock 登录，仅用于前端实训和状态管理演示，localStorage 中存储的是明文用户信息，不是生产环境安全认证方案，生产环境应使用 HttpOnly Cookie、JWT 等更安全的方案。

## 10. 遇到的问题与解决方法

**问题：登录状态刷新后丢失**

- 现象：最初实现登录功能后，登录成功时 Pinia 中状态正确，但按 F5 刷新页面后又回到未登录状态；
- 原因：初次实现时只在 Pinia 中保存了登录状态，没有将用户信息持久化到 localStorage，而 Pinia 的状态是存储在内存中的，页面刷新后内存数据被清空；
- 解决方法：按照 Day6 任务要求，在 `userStore` 中增加 `STORAGE_KEY` 常量，登录成功后调用 `localStorage.setItem(STORAGE_KEY, JSON.stringify(user))` 保存用户数据；新增 `restoreLogin()` 方法，在 `App.vue` 的 `onMounted` 钩子中调用，从 localStorage 读取数据并恢复到 Pinia；退出时调用 `localStorage.removeItem(STORAGE_KEY)` 清除持久化数据。修改后刷新页面登录状态正常保留。

**问题：window.alert 阻塞浏览器自动化测试**

- 现象：最初使用 `window.alert()` 显示"登录成功""发布成功"等提示，但在使用浏览器自动化工具进行功能走查时，alert 对话框会阻塞整个页面，导致后续自动化操作无法进行；
- 原因：`window.alert()` 是浏览器原生的模态对话框，会阻塞 JavaScript 主线程直到用户点击确定；
- 解决方法：创建了 Toast 通知 store（`src/stores/toast.ts`）和 Toast 组件（`src/components/Toast.vue`），将所有 `window.alert()` 替换为 `toastStore.success()` 或 `toastStore.error()`，Toast 消息自动 3 秒消失，不阻塞页面操作，体验更好。

## 11. 今日反思

通过补充注册登录功能，我对"用户状态""状态持久化"和"前端 Mock 认证边界"有了更深入的理解。

关于**用户状态**：用户的登录状态不是一个简单的布尔值，而是关联了一整套用户数据（id、name、college、grade 等），这些数据会被多个页面和组件消费——导航栏需要显示用户名，发布页需要读取用户名作为 publisher，个人中心需要展示完整的用户资料，这要求状态管理必须集中化、可预测，Pinia 正好提供了这样的能力。

关于**状态持久化**：我深刻理解了为什么前端状态需要持久化——内存是临时的，刷新即丢，而用户期望"登录一次，下次打开还是登录状态"这种基础体验。localStorage 是最简单直接的持久化方案，但使用时也要注意：不能存敏感信息、要处理 JSON 解析异常、退出时要同步清除。这次实践中还遇到了一个细节：restoreLogin 需要在 App 根组件的 onMounted 中尽早调用，否则子组件挂载时可能还读取不到用户信息。

关于**前端 Mock 认证边界**：这次实现让我清楚地认识到，前端 Mock 登录和真正的生产级认证有本质区别。我们在 db.json 里明文存密码、在 localStorage 里明文存用户信息、前端直接查询所有用户来校验密码——这些在生产环境中都是严重的安全问题。生产环境必须由后端进行密码校验、使用 HttpOnly Cookie 或短期 Token、密码要加盐哈希存储、要有 HTTPS 加密传输。但作为前端学习项目，这种 Mock 方式足够帮助我们理解"登录状态如何在前端流动"这个核心问题，避免一开始就陷入复杂的后端安全细节中。这也让我理解了"分阶段实现"的工程思维——先用简单方案跑通核心流程，再逐步替换为更安全、更完善的实现。
