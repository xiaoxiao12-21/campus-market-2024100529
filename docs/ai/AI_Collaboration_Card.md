## 3. 典型提示词

### 提示词 1：详情页设计

```
为四个主要功能（二手交易、失物招领、拼单搭子、跑腿委托）设计详情页，要求：
1. 每个详情页要有符合功能特点的布局和信息展示
2. 搭配与功能匹配的图片资源
3. 包含返回按钮和收藏功能
4. 保持与现有列表页的设计风格一致
5. 使用相同的状态组件（LoadingState、ErrorState、EmptyState）
```

**价值**：这个提示词明确了功能需求、设计约束和复用要求，AI 能够快速生成结构一致、风格统一的详情页代码，大幅提高开发效率。

### 提示词 2：错误状态处理优化

```
为所有列表页面添加统一的错误状态处理：
1. 当 Mock 服务未启动或请求失败时显示 ErrorState 组件
2. 错误组件需要显示友好的提示信息和重试按钮
3. 加载中显示 LoadingState 组件
4. 数据为空时显示 EmptyState 组件
5. 保持四个列表页的处理逻辑一致
```

**价值**：这个提示词强调了用户体验的一致性和错误恢复机制，帮助 AI 生成了完整的状态管理逻辑，而不是只处理成功路径。

## 4. AI 生成内容的问题

在使用 AI 辅助开发过程中，发现以下问题：

### 4.1 字段不贴合业务
- AI 初始生成的 Mock 数据字段过于通用（如电商场景的商品字段），需要调整为校园场景特有的字段（如宿舍区、教学楼、课程时间等）
- 部分字段命名不符合中文校园习惯，需要重新命名

### 4.2 代码过度复杂
- AI 有时会生成过于复杂的组件抽象，例如试图创建一个通用的"万能详情页"组件，反而增加了理解和维护成本
- 部分 TypeScript 类型定义过于繁琐，包含过多不必要的泛型和工具类型

### 4.3 引入了超出范围的功能
- AI 曾建议实现图片上传、富文本编辑器、实时聊天等超出课程范围的功能
- 部分生成的代码引入了项目中未安装的第三方库

### 4.4 组件抽象过度
- 某些情况下 AI 会将简单逻辑拆分成过多的小组件，导致组件层级过深，props 传递复杂
- 过度使用组合式函数，反而降低了代码可读性

### 4.5 生成内容无法直接运行
- 初始生成的代码存在 import 路径错误
- 部分 TypeScript 类型定义不匹配实际使用
- 缺少必要的依赖声明或样式引入

## 5. 我的判断与修改

### 5.1 业务字段调整
- 仔细审查 AI 生成的数据模型，将不符合校园场景的字段替换为更合适的字段
- 例如：将"商品库存"改为"商品成色"，将"配送方式"改为"交易地点"
- 统一了发布人信息格式，使用"专业+年级"的校园风格

### 5.2 简化代码结构
- 拒绝过度抽象，对于四个功能的详情页，采用独立组件而非一个通用组件
- 保持组件职责单一，避免为了"复用"而创建难以理解的抽象层
- 删除了不必要的 TypeScript 高级类型，保持类型定义简单直观

### 5.3 控制功能范围
- 严格按照课程任务要求，只实现规定的功能点
- 主动移除 AI 建议的图片上传、富文本编辑等复杂功能
- 对于登录功能，采用本地模拟方式，不接入真实后端认证

### 5.4 修复运行问题
- 逐一检查 import 路径，确保所有引用正确
- 运行类型检查，修复 TypeScript 类型错误
- 测试每个页面的实际运行效果，修复样式和交互问题
- 确保构建命令能够成功执行

### 5.5 状态管理优化
- 调整 Pinia Store 的设计，确保收藏状态在页面刷新后能够正确恢复
- 使用 localStorage 持久化用户登录状态和收藏数据
- 统一 Toast 提示组件，替代原生 alert 弹窗，避免阻塞浏览器自动化

## 6. 总结

AI 在本项目中显著提高了开发效率，主要体现在：

1. **快速搭建项目骨架**：AI 能够快速生成页面组件、路由配置、API 封装等样板代码，节省了大量重复劳动时间
2. **提供实现思路**：当遇到不确定如何实现的功能时，AI 可以给出多种实现方案供参考
3. **代码重构建议**：AI 能够识别代码中的重复模式，提出重构建议，帮助保持代码整洁
4. **文档生成**：AI 可以快速生成 README、注释等文档内容，减轻文档编写负担
5. **调试辅助**：当出现错误时，AI 可以帮助分析可能的原因并提供修复建议

然而，以下部分仍然必须由开发者自己理解和判断：

1. **业务逻辑理解**：AI 不了解具体的校园业务场景，需要开发者判断哪些字段和功能是真正需要的
2. **架构决策**：组件如何拆分、状态如何管理、代码如何组织，这些架构层面的决策需要开发者根据项目实际情况做出
3. **代码质量把控**：AI 生成的代码可能存在隐藏的 bug 或坏味道，需要开发者进行 code review 和测试验证
4. **功能范围控制**：AI 倾向于添加更多功能，开发者需要根据项目需求和时间约束，决定哪些功能应该实现，哪些应该舍弃
5. **最终验收测试**：AI 无法替代实际的功能测试，开发者需要亲自走查所有功能，确保项目能够正常运行和构建

总体而言，AI 是一个强大的编程助手，但它更像是一个"初级开发者"，需要有经验的开发者进行指导、审查和调整。合理使用 AI 可以大幅提高开发效率，但开发者自身的技术判断力和工程实践能力仍然是项目成功的关键。
